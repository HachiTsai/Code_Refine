# Spec: SEQ 順序控制、狀態機跳轉與製程意圖解碼協定 (SEQ Sequence & State-Machine Rationale Protocol)

## 🔍 第一階段：知識溯源與語義勾稽 (Phase 1: Discovery & Hooking)

在啟動正式分析前，AI 必須執行以下檢索程序，確保具備正確的解碼依據與跨模組深度：

### 層級負一：全場域訊號追蹤 (Global Signal Trace)
- **指令**：提取 `source_data` 中出現的所有外部輸入位址（如 `INxxxx`, `DHxxxx`, `DLxxxx`, `DOxxxx`）。
- **動作**：必須對該廠區的 `core.json` 目錄執行全量搜尋，找出這些位址的「定義源頭」（即該位址作為 `output` 的位置）。
- **目的**：確保跨模組鏈結的絕對精確性，嚴禁憑經驗猜測訊號源。

### 層級零：系統法典檢索 (System Codecs)
- **指令**：AI **必須優先讀取** 以下系統級規範文件，作為解讀 MPN 原始碼的唯一依據：
    1. **`20_Knowledge_Base\1_Hitachi_EX-N01A\0_Manual\2_SLC\0_Module_Definition\4_SEQ.md`**：定義 QP/QR/Pattern/Correction 優先級與狀態機運作流程。
    2. **`20_Knowledge_Base\1_Hitachi_EX-N01A\0_Manual\2_SLC\0_Module_Definition\4_SEQ_Always_Logic_Analysis.md`**：定義邏輯代號 1, 2, 3, A, B, C... 的布林代數轉換規則。

### 層級零點五：布林語法轉譯準則 (Boolean Codec Rules)
- **[重要] 運算子識別**：
    - **AND 邏輯**：原始碼中以 **「空格」** 表示。由於空格在 Markdown 表格中極難辨認且易產生歧義，AI 在生成與分析時 **必須統一轉換為 `&` 符號**。
    - **OR 邏輯**：以 **`+`** 符號表示。
    - **NOT 邏輯**：以 **`/`** 符號表示。
- **[目的]**：確保物理等式在文檔中具備高度的可讀性與可審計性。

### 層級一：顯性邏輯血緣 (Explicit Logic Lineage)
- **指令**：掃描全域的Twin目錄, `core.json` 中所有的輸出輸入訊號流。

- **動作**：SEQ 特別需要追蹤其驅動的外部 **[LPxxx]** (DDC 迴路) 與接收的 **[INxxxx]** 狀態位。
- **目的**：確保理解序列步驟對實體控制迴路的控制意圖。

### 層級二：語義標籤與物理勾稽 (Semantic Tags & Physical Context)
- **指令 1 [製程定位]**：從 `service_name` 提煉核心製程詞（如：`#反應控制`、`#清洗程序`）。
- **指令 2 [事實提取]**：讀取筆記頂層 `%% DCS_FACTS %%` 內的 Dataview 屬性（`Summary::`, `Importance::`）。
- **指令 3 [標籤聯動]**：搜尋廠區內所有標註 `**[核心詞]**` 或帶有相同 `#標籤` 的其他 SEQ 步驟。
- **目的**：識別不同設備之間是否存在相似的步序邏輯或同步交握。
- **訊號追溯**：(`Interlock::`)查詢**core.json**、**TWIN** 目錄 或**GID**

## 🌊 第二部分：統合邏輯鏈與安全防護解碼 (Agent Insight Callout)
- **指令**: AI 必須將以下維度的分析結果，統一彙整至頂層的 `> [!INFO]` Callout 區塊中，以提供直觀的工程導讀：

### 1. 准入判定與關鍵跳轉 (Shift Logic)
- **重點**: 深度拆解核心 SHIFT 邏輯（如 Logic 9「製程准入指標」）。解釋其物理條件（如：操作員確認、壓力穩定、負荷就緒）如何構成製程演進的「門禁」。

### 2. Fail-Safe 覆寫機制 (Correction Logic)
- **重點**: 分析 CORRECTION 邏輯如何動態覆寫 Pattern 輸出。明確指出其物理動機（例如：偵測到下游過載時強制攔截進料）。

### 3. 安全邊界與異常匯總 (Always Logic)
- **重點**: 拆解定義「異常 (ABNORMAL)」的超級或門（如 DO0004）。定義該設備的安全邊界，包含閥門回饋、系統跳機與過程極限。

### 4. 製程階段與時序演進 (Process / Timer)
- **重點**: 解析生產週期的劃分邏輯。說明「脈衝式合成」設計（如奇數步設定/偶數步執行）與安全延時防護的工程意圖。

## 🔄 第三部分：跨模組全域上下游數據流邏輯鏈結解析 (Cross-Module Linkage)
1. **DDC/SEQ/BLS 聯動**: 該 SEQ 驅動或監聽了哪些 **[LPxxx]**、**[USxxx]** 或 **[BLxxx]**？
    - **[精度要求]**: 必須精確對應。例如 `DH0007` 必須關聯至 **[LP007]**，嚴禁掛錯父迴路。
    - **[用語規範]**: 嚴禁使用「消費」等非工程用語。請統一使用 **「監聽」**、**「接收」** 或 **「引用」**。
2. **外部交握**: 必須明確標出信號源頭。若接收自 BLS，必須標出具體的編號（如：接收自 **[BL001]** 的彙整旗標）。
3. **全廠地位**: 該序列負責哪一個子系統（如：一號反應槽主序列）的流程總控。
4. **上下游關聯**: 是否存在其他 BLS/SEQ/DDC 邏輯塊與之形成聯鎖鏈（如 A 觸發 B，B 觸發 C）？該鏈條的安全意圖是什麼？
